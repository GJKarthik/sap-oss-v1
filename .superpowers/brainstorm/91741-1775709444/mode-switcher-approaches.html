Three Approaches: Mode Switcher in Shell Bar

How to bring Chat / Cowork / Training modes into your SAP AI workbench

A

Signal-Driven Mode State (Recommended)

How it works: A single activeMode signal in AppStore drives everything. The shell bar gets a segmented control (3 pills: Chat / Cowork / Training) replacing the product switcher. Switching modes updates the signal, which cascades via computed properties:

  • Nav: suggestedRoutes() computed from mode — highlights relevant routes, dims others
  • AI behavior: aiCapabilities() computed from mode — chat = conversational, cowork = plan-and-preview, training = autonomous execution
  • Context pill bar: Shows mode-specific quick actions

Pros

  • Fits your existing signal/computed architecture perfectly
  • Single source of truth — no state sync issues
  • Minimal new components (just the switcher + mode-aware computeds)

Cons

  • Every mode-aware component needs to read the signal
B

Route-Based Modes

How it works: Modes map to route prefixes: /chat/*, /cowork/*, /training/*. The shell bar switcher navigates between route groups. Each group has its own layout shell with mode-specific chrome.

  • Nav: Each route group defines its own nav links
  • AI behavior: Mode resolved from URL, injected via route data
  • Context pill bar: Different pill sets per route group

Pros

  • Deep-linkable — URLs encode the mode
  • Clean separation of concerns per mode

Cons

  • Breaks your "shared context" requirement — route changes reset state
  • Major refactor of existing 24+ routes
  • Duplicated layouts across mode shells
C

Service-Injected Mode Context

How it works: A new ModeService (injectable, not in store) manages mode state and exposes mode-aware tool registries. Components inject it to get current capabilities. The shell bar switcher calls modeService.setMode().

  • Nav: ModeService provides getRouteRelevance(route, mode)
  • AI behavior: ModeService provides tool registry per mode
  • Context pill bar: ModeService provides quick-action templates

Pros

  • Encapsulates all mode logic in one service
  • Easy to test mode transitions

Cons

  • Parallel state system alongside your existing signalStore — risks drift
  • Components need both AppStore and ModeService
  • Goes against your established @ngrx/signals pattern
