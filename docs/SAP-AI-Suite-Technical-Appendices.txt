APPENDIX A — GPU Performance Reference Table
================================================================================

Methodology
-----------
Peak memory-bound token throughput is calculated using the roofline model:

    tok/s = GPU_HBM_bandwidth_GB_s / model_weight_GB

where model_weight_GB = parameter_count × bytes_per_parameter.
  FP16  = 2 B/param   INT8 = 1 B/param   INT4 = 0.5 B/param

This formula is exact for a single-sequence decode step (the memory-bound
regime). Under continuous batching with many concurrent sequences the system
becomes compute-bound and can exceed this estimate. Dequantisation overhead
is modelled as a 15 % throughput penalty for INT8 and 25 % for INT4.

GPU hardware specifications used
---------------------------------
  T4 (PCIe, 16 GB GDDR6)        HBM BW:    300 GB/s
  A100 SXM (80 GB HBM2e)        HBM BW:  2,039 GB/s
  H100 SXM (80 GB HBM3)         HBM BW:  3,350 GB/s
  8 × H100 SXM NVLink            Agg BW: ~26,800 GB/s  (8 × 3,350)

Model weight sizes
-------------------
  7B  FP16  =  14.0 GB      7B  INT8  =  7.0 GB
  13B FP16  =  26.0 GB      13B INT8  = 13.0 GB
  70B FP16  = 140.0 GB      70B INT8  = 70.0 GB    70B INT4  = 35.0 GB

Roofline peak token throughput (tok/s, single-sequence decode)
---------------------------------------------------------------
                     T4          A100 SXM     H100 SXM    8×H100 NVLink
 7B  FP16            21             146          239          —
 7B  INT8            36 (×0.85)     247          482          —
 13B FP16            12              78          129          —
 13B INT8            20 (×0.85)     133          219          —
 70B FP16           N/A*             15†          24          191
 70B INT8           N/A*             25 (×0.85)†  41          273 (×0.85)
 70B INT4           N/A*             49 (×0.75)†  86          514 (×0.75)

Notes:
  * 70B FP16 = 140 GB; does not fit in T4 (16 GB) or A100 SXM (80 GB) VRAM.
    Requires tensor parallelism across multiple GPUs or quantisation.
  † Single A100 SXM can host 70B INT8 (70 GB) or 70B INT4 (35 GB).
    70B FP16 requires ≥ 2 × A100 SXM.
  N/A = model will not load in available VRAM.

Observed vLLM throughput on 7B FP16 under batch load (reported externally)
---------------------------------------------------------------------------
  Ollama sequential (single request)  ~41 tok/s     [Red Hat 2025, CITE NEEDED]
  vLLM 0.9.0 continuous batching     ~793 tok/s     [vLLM benchmark, CITE NEEDED]

The 793 tok/s figure is consistent with the roofline for a 7B model on
2 × A100 SXM under saturated batch load; it is not achievable for 70B models
on equivalent hardware without quantisation (roofline: 25–41 tok/s).

The document's claim that "a minimum of 46 ms per token on a T4, regardless
of the software" is only valid for a 7B FP16 model. The correct generalisation:
  min_latency_ms = (model_weight_GB / GPU_BW_GB_s) × 1000

Derived acceptance rate formula (from metrics.py):

What the ">90% acceptance on financial queries" claim requires
--------------------------------------------------------------
  1. Draft model fine-tuned or selected for SAP OData / financial vocabulary
     (field names: CompanyCode, FiscalYear, AccountingDocument, PostingDate, etc.)
  2. Evaluation set: N held-out financial queries; N must be stated.
  3. Hardware + model: T4 / A100 / H100; 7B or 70B target; FP16 or INT8.
  4. Comparison baseline: acceptance rate of the same draft model on
     general-purpose queries (expected ~60–80 %).
  5. Measurement: vllm:spec_decode_num_accepted_tokens /
                  vllm:spec_decode_num_draft_tokens over the evaluation run.

If the domain corpus genuinely covers SAP vocabulary, acceptance rates
of 85–92 % are plausible for 3-token MLP speculation; values above 95 %
are unusual and require explicit documentation.

--- F.1  Data subject and personal data field detection ---
Source: mangle-query-service/rules/governance.mg

  # Detect data subject entities from OData PersonalData annotations
  is_data_subject_entity(EntityType) :-
    entity_annotation(EntityType, "PersonalData.EntitySemantics", "DataSubject").

  # Pattern-based fallback for unannotated schemas
  is_data_subject_entity(EntityType) :-
    EntityType :> match("(?i)customer|employee|user|person|contact|patient|member").

  # Personal data field detection (annotated)
  is_personal_data_field(EntityType, FieldName) :-
    field_annotation(EntityType, FieldName, "PersonalData.IsPotentiallyPersonal", true).

  # Personal data field detection (pattern-based)
  is_personal_data_field(EntityType, FieldName) :-
    FieldName :> match("(?i)e?mail|phone|address|birth.*date|dob|ssn"),
    is_data_subject_entity(EntityType).

  # GDPR special-category sensitive data
  is_sensitive_data_field(EntityType, FieldName) :-
    FieldName :> match("(?i)health|medical|ethnic|religion|political|sexual|genetic|biometric|criminal").

--- F.2  Anonymisation rules ---
Source: mangle-query-service/rules/governance.mg

  # Sensitive fields must be anonymised in non-production environments
  must_anonymize(EntityType, FieldName) :-
    is_sensitive_data_field(EntityType, FieldName),
    environment("non-production").

  # Personal fields must be masked in audit logs and exports
  must_anonymize(EntityType, FieldName) :-
    is_personal_data_field(EntityType, FieldName),
    context("audit_log").

--- F.3  Access control ---
Source: mangle-query-service/rules/governance.mg

  # Data Protection Officer has unrestricted read access
  access_allowed(Query, "dpo", "role:dpo") :-
    is_data_subject_entity(_).

  # Regular users may access personal data only with verified consent
  access_allowed(Query, "user", "consent_verified") :-
    extract_entities(Query, EntityType, EntityID),
    consent_verified(EntityType, EntityID).

  # Deny access to GDPR special-category data for non-privileged roles
  !access_allowed(Query, Role, _) :-
    extract_entities(Query, EntityType, _),
    is_sensitive_data_field(EntityType, _),
    Role != "dpo",
    Role != "medical_staff".

--- F.4  GDPR data subject rights ---
Source: mangle-query-service/rules/governance.mg

  # Right of access — collect all personal data for a subject ID
  subject_access_request(SubjectID, PersonalData) :-
    is_data_subject_entity(EntityType),
    entity_has_id(EntityType, SubjectID, EntityID),
    collect_personal_data(EntityType, EntityID, PersonalData).

  # Right to erasure — identify records eligible for deletion
  subject_erasure_request(SubjectID, DataToDelete) :-
    is_data_subject_entity(EntityType),
    entity_has_id(EntityType, SubjectID, EntityID),
    !legal_hold(EntityType, EntityID),
    DataToDelete = {"entity_type": EntityType, "entity_id": EntityID}.

  # Right to portability — export in structured format
  subject_portability_request(SubjectID, ExportData) :-
    is_data_subject_entity(EntityType),
    entity_has_id(EntityType, SubjectID, EntityID),
    collect_personal_data(EntityType, EntityID, PersonalData),
    format_portable(PersonalData, ExportData).

  # Retention expiry detection
  data_retention_expired(EntityType, EntityID) :-
    field_annotation(EntityType, _, "PersonalData.EndOfBusinessDate", EndDateField),
    entity_field_value(EntityType, EntityID, EndDateField, EndDate),
    EndDate < current_date().

--- F.5  Governance resolution with audit gate ---
Source: mangle-query-service/rules/governance.mg

  # All resolution paths pass through audit gate and access check
  resolve_with_governance(Query, Answer, ResPath, Score) :-
    audit_required(Query, AuditReason),
    log_audit(Query, AuditReason),
    access_allowed(Query, current_user_role(), AccessReason),
    resolve(Query, RawAnswer, ResPath, RawScore),
    apply_anonymization(Query, RawAnswer, Answer),
    Score = RawScore.

  # Deny resolution if role check fails — returns literal "Access denied"
  resolve_with_governance(Query, "Access denied", "denied", 0) :-
    !access_allowed(Query, current_user_role(), _).

  in Zig (core broker + storage + transport) and Mojo (stream processing + SIMD
  embeddings) with Mangle for declarative routing and governance policy.
  It is not a wrapper around upstream Pulsar.
