\chapter{Cross-Specification Dependencies}
\label{ch:cross-spec}

This chapter documents how the Trial Balance Review specification integrates
with the other three SAP OSS specifications. All dependencies are traceable
to \texttt{docs/schema/registry.json} (v1.2.0).

% -----------------------------------------------------------------------------
\section{Architecture Position}
\label{sec:arch-position}

The Trial Balance specification occupies the \textbf{month-end close} position
in the four-domain architecture:

\begin{itemize}
  \item \textbf{Upstream}: Arabic AP (posted AP records feed variance explanations)
  \item \textbf{Downstream}: FORTM reporting (signed review packs)
  \item \textbf{Horizontal}: Regulations (governance wrapper for all AI stages)
  \item \textbf{Training}: Simula (consumes TB schemas for training data generation)
\end{itemize}

\section{Regulations Integration}
\label{sec:reg-integration}

The TB pipeline is wrapped by the Regulations domain's MAS MGF framework:

\subsection{Pillar 1: Assess \& Bound Risks}

\begin{itemize}
  \item MCP tool allow-list defined in Chapter~\ref{ch:implementation}
  \item Reversibility: Commentary drafts are suggestions, not final postings
  \item Autonomy bounds: Material variances require human approval
\end{itemize}

\subsection{Pillar 2: Human Accountability}

Human approval checkpoints (see Regulations Ch.3):
\begin{itemize}
  \item DP-001: Material variance detected ($>$30\% MoM)
  \item DP-002: Journal posting confirmation
  \item DP-003: Further investigation required
  \item DP-004: Third-party confirmation pending
  \item DP-005: Unexplained variance escalation
\end{itemize}

\subsection{Pillar 3: Technical Controls}

\begin{itemize}
  \item Drift monitoring: Commentary quality tracked per Regulations Ch.8
  \item Anomaly detection accuracy: False positive rate monitored
  \item Conformance testing: AI Verify hooks per Regulations Ch.6
\end{itemize}

\subsection{Pillar 4: End-User Responsibility}

\begin{itemize}
  \item Financial controller training curriculum per Regulations Ch.9
  \item Automation bias awareness for variance approval decisions
\end{itemize}

\section{Simula Integration}
\label{sec:simula-integration}

The Simula Training Data Framework consumes TB schemas:

\begin{itemize}
  \item \textbf{tb-extract.schema.json} $\rightarrow$ Training examples for account analysis
  \item \textbf{pl-extract.schema.json} $\rightarrow$ Training examples for P\&L commentary
  \item \textbf{variance-record.schema.json} $\rightarrow$ Variance explanation training
  \item \textbf{HANA metadata} $\rightarrow$ Schema extraction for ACDOCA/BSEG queries
\end{itemize}

See Simula specification Chapter 3 for schema extraction details.

\section{Arabic AP Integration}
\label{sec:arabic-integration}

Trial Balance consumes upstream data from Arabic AP:

\begin{itemize}
  \item Posted AP invoices appear as liability movements in TB extract
  \item AP records serve as evidence for variance explanations
  \item Vendor payment history supports commentary generation
  \item Cross-reference: See Arabic AP specification Chapter 5 (AP Workflow)
\end{itemize}
